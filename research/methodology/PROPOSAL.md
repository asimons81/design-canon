# Benchmark Methodology Proposal

## Design Canon Benchmark Factory — Evaluation Framework

### Status: Proposal v0.1

This document proposes a methodology for running, collecting, and evaluating paired frontend-generation benchmarks that compare three agent guidance conditions. It is a scaffold for structured experimentation, not a finalized procedure.

---

## 1. Purpose

The benchmark factory exists to answer one question empirically:

> **Does providing compiled, context-specific design policy produce measurably better frontend output than the alternatives, and if so, along which dimensions?**

The three conditions being compared are:

| Condition | Label | Description |
|---|---|---|
| A | **No guidance** | The agent receives only the benchmark brief (functional requirements, content, viewport constraints) with no design-specific instruction file |
| B | **Generic prompt** | The agent receives the benchmark brief plus a large (~4,000–8,000 token) monolithic "design system" prompt containing general-purpose frontend best practices (contrast, spacing, responsiveness, accessibility — but not specific to the benchmark category) |
| C | **Design Canon** | The agent receives the benchmark brief plus a compiled Design Canon policy document generated by `design-canon compile --profile <profile> --target design --output DESIGN.md` using the applicable profile |

Condition C is the experimental condition. Conditions A and B are baselines.

## 2. Benchmark Suite Overview

Fifteen benchmarks span three Design Canon profile categories:

| Profile | Benchmarks |
|---|---|
| `marketing` | B001 (SaaS landing), B007 (e-commerce product), B008 (pricing page), B009 (personal portfolio) |
| `product-app` | B002 (developer dashboard), B005 (app settings), B006 (mobile onboarding), B010 (analytics), B012 (auth flow), B013 (project board), B014 (mobile finance), B015 (legacy redesign) |
| `editorial` | B003 (editorial article), B004 (documentation site), B011 (public-service page) |

Each benchmark defines:
- Supplied content (text, data, descriptions — no external images or assets)
- Functional requirements
- Required components
- Interaction state expectations
- Desktop and mobile viewport dimensions
- Accessibility expectations
- Prohibited shortcuts
- Scoring categories (proposal — weights TBD)
- Expected deliverables
- Run-manifest fields
- Known ambiguity risks

## 3. Run Protocol

### 3.1 Prerequisites

Each run requires:
- A consistent agent environment (node, browser, model endpoint)
- A fixed Design Canon revision (commit hash)
- The benchmark fixture files (briefs, supplied content files)
- A capture tool (Puppeteer/Playwright or equivalent) for viewport screenshots
- An accessibility scanner (axe-core via @axe-core/cli or pa11y)
- The Design Canon linter (`design-canon lint`)

### 3.2 Per-Benchmark Procedure

1. **Prepare instructions**: Based on the condition (A/B/C), assemble the agent's instruction set:
   - Condition A: benchmark brief only
   - Condition B: benchmark brief + generic design prompt
   - Condition C: benchmark brief + compiled DESIGN.md from `design-canon compile --profile <profile>`
2. **Record environment metadata** (see Section 5)
3. **Submit prompt**: Send the assembled instructions to the model via the agent framework
4. **Capture output**: Save all generated source files
5. **Render**: Open the generated page in a headless browser at the specified viewport dimensions
6. **Capture screenshots**: Save desktop (1280×800) and mobile (375×667) screenshots
7. **Run linter**: Execute `design-canon lint` against the generated source
8. **Run accessibility scan**: Execute axe-core or equivalent against the rendered page
9. **Record runtime and token usage**: From the model provider's response metadata
10. **Store artifacts**: Commit all artifacts to the run's directory

### 3.3 Reproducibility Constraints

- Temperature: fixed (proposed: 0.7) for all runs within a comparison set
- System prompt: the agent's default system prompt (if any) should be identical across all three conditions within a comparison set
- Model version: recorded and fixed for the duration of one comparison set
- A single comparison set = 3 conditions × 15 benchmarks = 45 runs at minimum
- Multiple comparison sets across different models, model versions, or temperature settings are encouraged

## 4. Artifact Directory Structure

Each run produces a directory under `research/runs/`:

```
research/runs/
  └── <run-id>/
      ├── manifest.json              # All run-manifest fields
      ├── prompt.md                  # The full instruction sent to the model
      ├── instructions/              # Any instruction/policy files provided
      │   ├── brief.md               # The benchmark brief
      │   ├── design-policy.md       # (Condition C only) Compiled DESIGN.md
      │   └── generic-prompt.md      # (Condition B only) Generic prompt
      ├── source/                    # Generated source tree
      │   ├── index.html
      │   ├── style.css
      │   └── app.js (if applicable)
      ├── screenshots/
      │   ├── desktop-1280x800.png
      │   ├── mobile-375x667.png
      │   └── <state>-<viewport>.png # Additional state screenshots
      ├── lint-report.json           # design-canon lint output
      ├── a11y-report.json           # axe-core or pa11y output
      └── metadata.json              # Extended environment + timing info
```

## 7. Subjective Evaluation Categories (Proposal)

These categories require human judgment. The proposed method is blind comparison: evaluators see rendered screenshots without knowing which condition produced them.

### 7.1 Blind Comparison Protocol

1. Two screenshots (desktop and mobile) from run A, B, and C are shown side by side in a random order
2. Evaluator is not told which is which
3. Evaluator answers a set of preference questions (see below)
4. Multiple evaluators score independently
5. Results are aggregated per benchmark

### 7.2 Proposed Preference Questions

Each evaluator answers for each pair:

| Question | Options |
|---|---|
| "Which interface communicates its purpose more clearly at first glance?" | A / B / C / No preference |
| "Which has better visual hierarchy and scanability?" | A / B / C / No preference |
| "Which has more appropriate typography for its content?" | A / B / C / No preference |
| "Which uses color more effectively (not just more color)?" | A / B / C / No preference |
| "Which feels more trustworthy and professional?" | A / B / C / No preference |
| "Which would you rather use for the task described?" | A / B / C / No preference |
| "Rate the overall visual quality of each." | 1–5 scale per condition |
| "Rate the likely real-world usability of each." | 1–5 scale per condition |
| "Rate the novelty / non-template quality of each." | 1–5 scale per condition |

### 7.3 Evaluator Pool

- Minimum 3 evaluators per benchmark comparison set (proposed)
- Evaluators should include: at least one experienced UI/UX designer, at least one developer, and at least one non-designer stakeholder
- Evaluators must not know which condition produced which screenshot
- Evaluators should not be the person who conducted the runs

### 7.4 Scoring Rubric (Proposal — Weights Not Final)

The following categories are proposed for scoring. Weights are deliberately **not** assigned here and are left for review and calibration.

| Category | Type | Proposed method |
|---|---|---|
| Visual hierarchy | Subjective | Blind comparison preference + 1–5 rating |
| Typography | Subjective | Blind comparison preference |
| Color usage | Subjective | Blind comparison preference |
| Component consistency | Objective + Subjective | Lint (repeated token usage) + blind comparison |
| Responsiveness | Objective | Overflow detection + blind comparison of mobile layout |
| Interaction feedback | Objective | CSS state selector presence + blind comparison of hover/focus quality |
| Content structure | Objective | Heading hierarchy check (h1 → h2 → h3) |
| Accessibility | Objective | axe-core violation counts |
| Layout density | Objective + Subjective | Content-to-viewport ratio + blind comparison |
| Generic-default avoidance | Subjective | Blind comparison (evaluator asked to identify "AI-generated tell" patterns) |
| Overall preference | Subjective | "Which would you rather use?" aggregated across evaluators |

## 8. Proposed Analysis and Reporting

### 8.1 Per-Benchmark Report

Each benchmark produces a structured report containing:
- Three run manifests (A, B, C)
- Objective metrics comparison table
- Screenshot pair
- Lint and a11y summary
- Human preference tallies

### 8.2 Aggregated Report

Across all 15 benchmarks:
- Which condition wins most often on each objective metric (e.g., fewest a11y violations, lowest token usage)
- Which condition wins most often in blind human preference (overall and per category)
- Breakdown by profile (marketing, product-app, editorial) — does Design Canon perform better for certain surface types?
- Cost-vs-quality analysis (how much additional quality does each condition buy per dollar of inference cost?)

### 8.3 Statistical Approach (Proposal)

- **Objective metrics**: Report mean, median, min, max, and standard deviation per condition across all benchmarks. Use paired comparisons where possible (same benchmark, different condition).
- **Subjective metrics**: Report preference frequencies. For 1–5 ratings, report mean and distribution. Cohen's kappa to measure inter-evaluator agreement.
- **Significance testing**: Use Wilcoxon signed-rank or bootstrap methods for paired comparisons. Do not rely on p-values alone; report effect sizes.
- **Visualization**: Paired bar charts, spider/radar charts per category, and preference matrices.

## 9. Threats to Validity

### 9.1 Internal Validity

| Threat | Mitigation |
|---|---|
| Model nondeterminism (temperature > 0) | Run each condition × benchmark × model combination multiple times (proposed: 3 runs minimum) |
| Prompt engineering skill differences | The generic prompt (Condition B) must be authored by an experienced frontend engineer and reviewed by a second engineer before use |
| Order effects in blind comparison | Randomize presentation order per evaluator |
| Evaluator fatigue | Limit evaluations to 5 benchmarks per session; break across multiple sessions |
| Linter rule gaps | Design Canon's linter is a heuristic — it may miss violations. Document false negatives per run |

### 9.2 External Validity

| Threat | Mitigation |
|---|---|
| Model-specific results | Run across multiple model families (proposed: at least GPT-4o, Claude Sonnet, Gemini) |
| Single-agent-framework results | Run with at least two agent frameworks (proposed: Codex CLI and Hermes Agent) |
| Artificial benchmark content | Briefs use realistic (but fictional) content; real-world interfaces may differ |
| Prohibited shortcuts may not reflect real workflows | The prohibitions are designed to isolate design skill from library knowledge — clearly documented as a research constraint |

### 9.3 Construct Validity

| Threat | Mitigation |
|---|---|
| "Better" is subjective | Separate objective and subjective metrics clearly; report both |
| Blind comparison exposes visual differences only | Source quality, maintainability, and runtime performance are not captured — document as scope limits |
| Accessibility scan misses contextual issues | Automated scans catch ~30–50% of WCAG issues; manual a11y review is future work |

## 10. Proposed Scoring Weights (Do Not Finalize)

**This section is intentionally left as a template for review.**

The following weighting schema is proposed for discussion. It is **not finalized** and must be reviewed and calibrated by the project maintainers before adoption.

```
Category                    Proposed Weight    Rationale
─────────────────────────────────────────────────────────
Visual hierarchy            25%                Primary design goal
Accessibility               20%                Non-negotiable baseline
Typography                  15%                Core to readability
Responsiveness              10%                Cross-viewport requirement
Color usage                 10%                Brand and communication
Component consistency       10%                Reusable output quality
Interaction feedback         5%                Detail polish
Generic-default avoidance    5%                Distinctive from AI default
```

**Important**: This weighting is a starting point for discussion. It must not be used for final scoring until reviewed, debated, and approved. Different stakeholder groups (designers, engineers, accessibility specialists) may reasonably assign different weights. Consider conducting a weighting calibration exercise before the first official run.

## 11. Open Questions for Review

1. **Number of evaluators**: Is 3 per comparison set sufficient, or should targeting 5+ for statistical power?
2. **Evaluator expertise**: Should evaluators be screened for design experience, or should we include a representative mix including non-designers?
3. **Model sampling**: Should we run each condition once or multiple times (proposed: 3×) and report the range?
4. **Generic prompt content**: Who authors the generic prompt, and is there a single canonical generic prompt or one per benchmark category?
5. **Scoring weights**: The proposed weights in Section 10 — do they reflect the project's priorities?
6. **Pairwise vs. three-way comparison**: Should evaluators see all three conditions at once (A/B/C) or only pairwise (A vs. B, B vs. C, A vs. C)?
7. **Source inspection**: Should evaluators review source code in addition to screenshots, or only visual output?
8. **Minimum runtime**: Should we impose a minimum time budget per run (e.g., "no more than 5 minutes wall-clock") to prevent models from over-optimizing?
9. **Baseline condition**: Should Condition A include the same word-count budget for the brief as Conditions B and C, or is the brief-as-only-instruction intentionally shorter?
10. **Prohibited shortcuts enforcement**: How should violations of the "prohibited shortcuts" section (e.g., using Tailwind) be handled — mark as incomplete, flag for manual review, or exclude?
