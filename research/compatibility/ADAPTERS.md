# Adapter Recommendations

> **Research date:** 2026-07-18
> **Status:** Proposal — do NOT implement adapters without maintainer review.
> Each claim tagged with confidence level. Conflicting/undocumented behavior flagged.

---

## Guiding Principles

1. **One source, many targets** — Design Canon compiles once (from profiles, rules, schemas); adapters reformat the compiled output for each tool's expected format.
2. **Least surprise** — Use whatever filename, location, and format the tool already expects natively.
3. **Zero runtime dependencies** — Each adapter should be a simple template transform (handlebars, EJS, or even string concatenation).
4. **Provenance marking** — Every generated file should include a machine-readable comment indicating it was produced by Design Canon (to aid debugging).
5. **Do not guess** — If a tool's format requirements are undocumented, flag that instead of inventing a format.

## 1. Hermes Agent Adapter

**Confidence level:** `verified fact` — well-documented format

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `AGENTS.md` (project context, auto-loaded) |
| **Alt filename** | `skills/design-canon/SKILL.md` (on-demand skill, loaded via `/design-canon`) |
| **Format** | Plain markdown for `AGENTS.md`; Markdown with YAML frontmatter for `SKILL.md` |
| **Placement** | Project root for `AGENTS.md`; `~/.hermes/skills/design-canon/SKILL.md` for skill deployment |
| **Frontmatter (skill)** | `name: design-canon`, `description: Enforces Design Canon policies for <profile-name>`, optional `category: design` |

### `AGENTS.md` output format (preferred)

Since Hermes auto-loads `AGENTS.md` from the project root and it's **higher priority than `CLAUDE.md` and `.cursorrules`**, this is the simplest and most effective delivery mechanism:

```markdown
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

## Design Rules

- [compiled design rules as plain markdown]
```

No frontmatter needed. Keep under 20,000 chars (Hermes' `context_file_max_chars` default).

### `SKILL.md` output format (alternative)

For on-demand use (team members who want design rules available but not always loaded):

```markdown
---
name: design-canon-<profile-slug>
description: Enforces the <profile-name> Design Canon policies
version: 1.0.0
metadata:
  hermes:
    category: design
    tags: [design-system, ui, branding]
---

# Design Canon — <profile-name>

[compiled design rules as markdown]

## When to Use

When implementing or reviewing UI components, layouts, or visual elements.

## Verification

Run `design-canon check` to verify adherence.
```

### Notes
- **Hermes already has a `skills/design-canon/SKILL.md` in the repo** — the adapter should produce content compatible with that skill's invocation pattern.
- The `AGENTS.md` approach is simpler and leverages auto-loading.
- The skill approach gives users explicit control over when the canon is activated.
- **Consider generating both** — `AGENTS.md` for auto-loading, and update the existing `SKILL.md` for on-demand use.

## 2. OpenAI Codex (CLI) Adapter

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `AGENTS.md` |
| **Format** | Plain markdown (no frontmatter) |
| **Placement** | Project root |
| **Cross-tool compat** | Also serves Claude Code, Cursor, Windsurf, and generic AGENTS.md agents |
| **Size guidance** | Keep concise — Codex context window varies by model |

### Output format

```markdown
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

## Design System

[design tokens and rules as plain markdown sections]

## Code Style

[naming conventions, spacing, component patterns]
```

### Notes

- **No frontmatter needed** — Codex treats the entire file as plain markdown instructions.

## 3. Claude Code Adapter

**Confidence level:** `verified fact` — well-documented on code.claude.com

### Recommended output

| Field | Value |
|-------|-------|
| **Primary filename** | `CLAUDE.md` (Claude Code's native format) |
| **Alt filename** | `AGENTS.md` (cross-tool, loaded by Claude Code but with lower specificity) |
| **Format** | Plain markdown (no frontmatter for CLAUDE.md) |
| **Placement** | Project root |
| **Path-scoped rules** | `.claude/rules/design-canon.mdc` with `paths` frontmatter for modular rules |

### Output format (CLAUDE.md)

```markdown
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

## Design Rules

[compiled design rules as markdown sections]

## Component Patterns

[component-specific guidance]
```

### Path-scoped rules alternative

For larger canons, consider splitting into `.claude/rules/*.md` files:

```markdown
---
paths: "**/*.{tsx,jsx,vue,svelte}"
---

# Design Canon — Component Rules

[component-specific design rules]
```

### Notes
- Claude Code walks **up the directory tree** collecting all `CLAUDE.md` files — a root-level file covers the whole project.
- Subdirectory `CLAUDE.md` files are **lazily loaded**. For large monorepos, generate per-package `CLAUDE.md` files that load only when relevant.
- `CLAUDE.md` is loaded **in full** regardless of length, but shorter files (under ~200 lines) produce better adherence.
- Project-root `CLAUDE.md` survives `/compact` — nested ones don't (they re-load lazily).
- **Undocumented behavior**: Whether Claude Code merges `AGENTS.md` and `CLAUDE.md` or prefers one over the other is unclear. **Recommendation:** Generate only `CLAUDE.md` to avoid ambiguity.

## 4. Cursor IDE Adapter

**Confidence level:** `verified fact` — well-documented on cursor.com/docs

### Recommended output

| Field | Value |
|-------|-------|
| **Primary filename** | `.cursor/rules/design-canon.mdc` (modular, preferred) |
| **Alt filename** | `.cursorrules` (legacy, deprecated but supported) |
| **Format** | Markdown with YAML frontmatter for `.mdc`; plain markdown for `.cursorrules` |
| **Placement** | `.cursor/rules/` directory (project root) |
| **Frontmatter (.mdc)** | `description` (required), `globs` (optional, for file-scoped rules) |

### `.mdc` output format (preferred)

```yaml
---
description: Design Canon <profile-name> design rules — applies to UI files
globs: "*.{css,html,jsx,tsx,vue,svelte}"
---
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

[design rules as markdown]
```

### Multi-file approach (recommended for complex canons)

Split the canon into focused `.mdc` files, one per concern:

| File | Globs | Content |
|------|-------|---------|
| `.cursor/rules/design-canon-tokens.mdc` | `*.{css,scss,less}` | Design tokens (colors, spacing, typography) |
| `.cursor/rules/design-canon-components.mdc` | `*.{tsx,jsx,vue,svelte}` | Component patterns and conventions |
| `.cursor/rules/design-canon-layout.mdc` | `*.{tsx,jsx,css}` | Layout rules and spacing |

### Notes
- `.cursorrules` is deprecated but still read. Prefer `.cursor/rules/*.mdc`.
- AGENTS.md at project root is also supported by Cursor — if you generate `AGENTS.md` for other tools, Cursor will also read it.
- `.mdc` files without a `globs` field apply to all files. Use `globs` for focused rules.
- Refresh requires **IDE reload** (`Developer: Reload Window`).

## 5. Windsurf IDE Adapter

**Confidence level:** `research finding` — documentation is sparse; product recently rebranded

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `.windsurfrules` (legacy but still works) |
| **Alt filename** | `.devin/rules/design-canon.md` (new path post-rebrand) |
| **Format** | Plain markdown (best assumption given sparse docs) |
| **Placement** | Project root |
| **Frontmatter** | Not documented — assume none needed |

### Output format

```markdown
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

[design rules as markdown]
```

### ⚠️ Warnings

| Issue | Detail |
|-------|--------|
| **Sparse documentation** | Official Windsurf docs do not clearly specify the rule file format, frontmatter requirements, or loading behavior. |
| **Product rebrand** | Windsurf → Devin Desktop (June 2026). `.devin/rules/` is the new path, but `.windsurf/rules/` is kept as fallback. |
| **Format uncertainty** | Whether `.devin/rules/*.md` files require YAML frontmatter (like Cursor's `.mdc`) is undocumented. |
| **Testing gap** | No publicly available test confirms which formats actually load. |

### Recommendation

1. Generate `.windsurfrules` as plain markdown for backward compatibility (easiest path).
2. Also generate `AGENTS.md` — Windsurf likely reads it as a cross-IDE standard.
3. **Flag** the Windsurf adapter as experimental until official documentation clarifies the format.
4. Revisit post-rebrand when Devin Desktop documentation stabilizes.

## 6. Generic AGENTS.md Adapter

**Confidence level:** `verified fact` — the AGENTS.md standard is well-defined at agents.md

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `AGENTS.md` |
| **Format** | Plain markdown (no frontmatter) |
| **Placement** | Project root |
| **Fallback placement** | `.github/AGENTS.md` (GitHub UI also recognizes this) |

### Output format

```markdown
# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

## Design Rules

[compiled design rules as markdown sections]

## Conventions

[naming, patterns, structure guidance]
```

### Notes
- This is the **most portable format** — recognized by Codex, Claude Code, Cursor, Windsurf, Cline, Copilot, Gemini CLI, Aider, Zed, Warp, Goose, OpenCode, Jules, Factory, and more.
- **No frontmatter needed** — the standard explicitly says "no required fields."
- Recommended as the **default adapter** when the target tool is unknown.
- Nested `AGENTS.md` files in subdirectories are supported by most tools — useful for monorepos.
- The standard is stewarded by the Agentic AI Foundation under the Linux Foundation, ensuring long-term stability.

## 7. DESIGN.md (Google Labs Spec) Adapter

**Confidence level:** `verified fact` — the spec is published and well-defined

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `DESIGN.md` |
| **Format** | YAML frontmatter (design tokens) + Markdown body (design rationale) |
| **Placement** | Project root |
| **Frontmatter required** | Yes — at minimum `name` and a `colors` palette with at least `primary` |

### Output format

```yaml
---
version: alpha
name: <profile-name>
description: <profile-description>
colors:
  primary: "<hex>"
  secondary: "<hex>"
  tertiary: "<hex>"
  neutral: "<hex>"
typography:
  h1:
    fontFamily: <font>
    fontSize: <size>
    fontWeight: <weight>
    lineHeight: <line-height>
  body-md:
    fontFamily: <font>
    fontSize: <size>
    fontWeight: <weight>
    lineHeight: <line-height>
spacing:
  xs: 4px
  sm: 8px
  md: 16px
  lg: 32px
  xl: 64px
rounded:
  sm: 4px
  md: 8px
  lg: 12px
  full: 9999px
components:
  button-primary:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.neutral}"
    rounded: "{rounded.md}"
    padding: 12px
---
# Design Canon — <profile-name>

## Overview

[brand personality, design philosophy]

## Colors

[human-readable color descriptions]

## Typography

[type scale rationale]

## Layout & Spacing

[layout model description]

## Components

[component-level guidance]

## Do's and Don'ts

[practical guidelines and pitfalls]
```

### Token reference syntax

The DESIGN.md spec uses `{path.to.token}` syntax for cross-referencing values:

```yaml
components:
  button-primary:
    backgroundColor: "{colors.primary}"
    typography: "{typography.body-md}"
```

### Notes
- This is Design Canon's **native design-output format** — what the canon compiles FROM profiles and rules TO a standard DESIGN.md.
- The spec is in "alpha" version. Expect evolution.
- Not all coding agents natively recognize DESIGN.md yet — supported tools include Claude Code, Cursor, Kiro, and Windsurf (per designmd.app).
- **Best practice**: Generate both `DESIGN.md` (for design-aware tools) AND `AGENTS.md` (for universal compatibility).
- Section order matters: Overview → Colors → Typography → Layout → Elevation → Shapes → Components → Do's and Don'ts.
- Unknown sections are preserved without error; duplicate sections cause rejection.

## 8. Agent Skills Adapter

**Confidence level:** `verified fact` for Hermes; `research finding` for other tools

### Recommended output

| Field | Value |
|-------|-------|
| **Filename** | `SKILL.md` |
| **Format** | Markdown with YAML frontmatter |
| **Placement** | `skills/design-canon/` relative to project root, or `~/.hermes/skills/<category>/design-canon/` |
| **Frontmatter required** | `name` and `description` |

### Output format

```markdown
---
name: design-canon-<profile-slug>
description: Enforces Design Canon <profile-name> design policies
version: 1.0.0
metadata:
  hermes:
    category: design
    tags: [design-system, ui, branding]
---

# Design Canon — <profile-name>

> Generated by Design Canon v<version>. Do not edit manually.

## When to Use

- When creating new UI components or pages
- When modifying existing visual elements
- When reviewing pull requests with UI changes
- When asked about design conventions

## Procedure

1. Review the design rules below before implementing UI
2. Apply the Design Canon during implementation
3. Verify output against the rules

## Design Rules

[compiled design rules as markdown sections]

## Verification

Run `design-canon check [files...]` to verify adherence.
```

### Notes
- Primarily useful for **Hermes Agent** — other tools may have incompatible skill systems.
- If Hermes Agent users want auto-loading, prefer the AGENTS.md adapter instead.
- The skill approach gives explicit on-demand control via `/design-canon` slash command.
- External skill directories (`skills.external_dirs` in config.yaml) can share these across projects.

## Cross-Tool Adapter Mapping

### Recommended adapter by scenario

| Scenario | Primary Adapter | Secondary Adapter | Rationale |
|----------|----------------|-------------------|-----------|
| **Don't know the tool** | AGENTS.md (generic) | DESIGN.md | AGENTS.md is the most portable format |
| **Hermes Agent user** | AGENTS.md (project root) | SKILL.md (skill) | Auto-loads via AGENTS.md; on-demand via skill |
| **Claude Code user** | CLAUDE.md | .claude/rules/*.md | CLAUDE.md is the native format with best support |
| **Codex CLI user** | AGENTS.md | N/A | Codex's native format |
| **Cursor user** | .cursor/rules/*.mdc | AGENTS.md | Modular .mdc rules + AGENTS.md fallback |
| **Windsurf/Devin user** | .windsurfrules | AGENTS.md + .devin/rules/ | Legacy + new path (experimental) |
| **DESIGN.md consumer** | DESIGN.md | AGENTS.md | DESIGN.md is the spec format |
| **Monorepo** | Per-package AGENTS.md/CLAUDE.md | Root AGENTS.md | Per-package files leverage nested discovery |

## Undocumented & Conflicting Behaviors — Adapter Impact

| Issue | Affected Tools | Impact on Adapter |
|-------|---------------|-------------------|
| **Windsurf rule format** unknown | Windsurf/Devin Desktop | Adapter must assume plain markdown until docs clarify. Flag as experimental. |
| **Claude Code AGENTS.md + CLAUDE.md merge** | Claude Code | Generate only one file (CLAUDE.md) to avoid confusion. |
| **Codex AGENTS.override.md** hierarchy | Codex CLI | Not relevant for project-local Design Canon output. |
| **Cursor AGENTS.md integration depth** | Cursor | AGENTS.md is known to work but may not have full feature parity with .mdc rules. |
| **DESIGN.md automatic loading** | Various | The spec doesn't define auto-loading behavior. Tools may require explicit user instruction to read DESIGN.md. |
| **Skills format portability** | Non-Hermes tools | The agentskills.io standard is emerging; other tools may have incompatible expectations. |

## Implementation Recommendations

### Priority order for building adapters

1. **AGENTS.md** (generic) — simplest, most portable, covers most tools
2. **CLAUDE.md** — well-documented, Claude Code's native format
3. **DESIGN.md** — Design Canon's native output format, Google Labs spec
4. **SKILL.md** — for Hermes Agent on-demand loading
5. **.cursor/rules/*.mdc** — for Cursor users who want granular rules
6. **.windsurfrules / .devin/rules/** — experimental, lowest priority

### Shared concerns across all adapters

1. **Provenance header**: Every generated file should include:
   ```markdown
   > Generated by Design Canon v<version> from profile <profile-name>.
   > Do not edit manually — changes will be overwritten.
   ```
2. **Size management**: Canon profiles may produce large outputs. Consider:
   - Splitting into per-concern files (tokens, components, layout)
   - Using tool-specific modular formats (`.claude/rules/`, `.cursor/rules/`)
   - Providing a "lite" option that excludes verbose rationale
3. **Refresh guidance**: Document that the user must restart/reload their tool after regenerating.
4. **Testing**: Each adapter output should be verifiable with the target tool (e.g., `claude /context` to confirm CLAUDE.md loaded).
